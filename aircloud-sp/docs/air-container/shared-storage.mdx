A persistent storage layer that retains data even when a container is restarted or stopped. Within a single project, volumes can be shared across multiple containers or used simultaneously by multiple replicas.

## What you can do with persistent volumes

Persistent volumes are available throughout the deployment lifecycle with greater flexibility.

- **Autoscaling support**: Volumes can be attached to deployments that use autoscaling and multiple replicas.
- **New or existing volume at creation**: When creating a deployment, you can create a new volume or reuse an existing one.
- **Share across multiple deployments**: A single volume can be connected to multiple deployments. When selecting a volume, the console shows which deployments are already using it.
- **Change while stopped**: While a deployment is stopped (Pending), you can activate, deactivate, or swap the attached volume.

## How to use

### 1. Create a new volume

When creating a container, you can create and attach a new persistent volume. Maximum volume size is 30 GB.

![Create new volume](/images/first_step.png)

### 2. Attach an existing volume

You can attach a volume already in use by another container to share it across deployments. When selecting a volume, the console lists the deployments currently using it.

![Attach existing volume](/images/second_step.png)

### 3. Share one volume across multiple replicas

Multiple replicas within a single container can use the same volume simultaneously. This also works for autoscaling-enabled deployments.

![Shared volume across replicas](/images/shared_storage.png)

### 4. Change the volume while stopped

When a container is in **Pending (stopped)** state, you can modify volume settings:

- **Deactivate** or **reactivate** the attached volume.
- **Swap** the attached volume for a different one.

Changes take effect when the container is started again.
